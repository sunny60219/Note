# Coding Standard

本文件用於統一本專題之 C# 程式碼撰寫規範，以提升可讀性、維護性與團隊協作效率。本規範可隨專題進行中持續調整與補充。

## 一、命名規範 Naming Convention

### 1. 變數與參數（camelCase）
- 第一個單字小寫
- 第二個（含）之後的單字首字母大寫

```csharp
int totalCount;
string userName;
bool isCompleted;

void CreateOrder(int orderId, string userName)
```

### 2. 方法與類別（PascalCase）
- 每個單字的首字母皆為大寫

```csharp
class OrderService
{
    public void CreateOrder()
    {
    }
}
```

### 3. 常數（Constant）
- 使用 PascalCase 命名
- 命名需具備語意，避免使用魔法數字

```csharp
const int MaxRetryCount = 3;
const string DefaultRoleName = "User";
```

## 二、程式碼格式 Formatting

### 1. 自動格式化
- 每次完成 C# 程式碼撰寫後，請使用 Visual Studio 快捷鍵 Ctrl + K + D 進行自動格式化

```text
Ctrl + K + D
```

### 2. 空格與排版規則
- 運算子前後需保留空格
- 參數之間以「逗號 + 空格」分隔
- 符號與變數之間需保留適當間距，以提升閱讀性
- 可用 1. 自動格式化 達成此格式

```csharp
if (count > 0 && isValid)
{
    totalPrice = unitPrice * count;
}
```

### 3. 大括號風格
- 使用 Allman Style（C# 預設）
- 大括號需獨立成行，不可與程式碼同行

```csharp
if (isActive)
{
    DoSomething();
}
```

## 三、可讀性與結構 Readability

### 1. 方法職責單一
- 一個方法僅負責一件事情
- 若方法內容過長，請考慮拆分為多個方法

```csharp
public void ProcessOrder()
{
    ValidateOrder();
    CalculatePrice();
    SaveOrder();
}
```

### 2. 命名優先於註解
- 優先使用清楚且具語意的命名
- 避免僅用註解補足命名不足的情況

```csharp
// 不建議
int x;

// 建議
int totalUserCount;
```

## 四、Razor (.cshtml) 元件樣式一致性規範

為確保專案 UI 一致性與範本完整性，所有新建立之 Razor 元件，**必須沿用範本既有樣式 class，不可自行建立相似但不同名稱之 class。**

### 1️⃣ 新增元件標準流程（必須遵守）

當需新增 UI 元件（如下拉選單、按鈕、導覽列項目、卡片、表單欄位等）時，請依照以下步驟進行：

#### Step 1：建立功能正確的 Razor 元件

請使用 Tag Helper 標準寫法：

```html
<a class="dropdown-item"
   asp-controller="Shop"
   asp-action="CatalogElectronics">
    Electronics
</a>
```

不得使用：

```html
<a href="/Shop/CatalogElectronics">Electronics</a>
```

或：

```html
<a href="#">Electronics</a>
```

---

#### Step 2：搜尋範本中相同類型元件

在現有範本 `.cshtml` 檔案中：

- 使用 `Ctrl + F`
- 搜尋相關關鍵字，例如：
  - dropdown-item
  - nav-link
  - btn
  - form-control
  - card
  - modal

必須找到「功能相同或結構相似」的現有寫法。

---

#### Step 3：套用範本既有 class

範例（範本原始寫法）：

```html
<a class="dropdown-item"
   asp-controller="Shop"
   asp-action="CatalogElectronics">
    Electronics
</a>
```

若新增新選項：

```html
<a class="dropdown-item"
   asp-controller="Shop"
   asp-action="CatalogClothes">
    Clothes
</a>
```

禁止自行創建：

```html
<a class="my-dropdown-item"
   asp-controller="Shop"
   asp-action="CatalogClothes">
    Clothes
</a>
```

---

### 2️⃣ 禁止事項

- 不可自行新增樣式 class（未經討論）
- 不可使用 inline style
- 不可直接修改 theme.min.css
- 不可自行複製 Bootstrap 結構並改名

---

### 3️⃣ 若無對應樣式時

若搜尋後確實無範本對應樣式：

1. 先於團隊提出討論
2. 決議是否新增全域樣式
3. 統一新增於專案指定 CSS 檔案
4. 更新 Coding Standard 文件

## 五、Git 分支命名規範 Git Branch Naming

為確保分支用途清楚、責任歸屬明確，所有新建立之功能分支，必須依照以下命名規則。

---

### 1️⃣ 分支命名格式

```
ControllerName/ActionName/DeveloperName
```

範例：

```
Account/Register/Owen
Shop/CatalogElectronics/Sunny
Order/Create/Jack
```

說明：

- ControllerName：對應專案中的 Controller 名稱（不含 "Controller" 字尾）
- ActionName：對應實作之 Action 方法名稱
- DeveloperName：負責該功能之開發者名稱

---

### 2️⃣ 建立分支的來源規則

所有新分支 **必須從 main 分支建立**。

標準流程：

```bash
git checkout main
git pull origin main
git checkout -b ControllerName/ActionName/DeveloperName
```

範例：

```bash
git checkout main
git pull origin main
git checkout -b Account/Register/Owen
```

---

### 3️⃣ 禁止事項

- 不可從其他功能分支建立新分支
- 不可在未更新 main 的情況下建立分支
- 不可使用無意義命名（如 test1、newbranch、fix 等）

---

### 4️⃣ 設立此規範之目的

- 明確對應功能與負責人
- 提升 Git log 可讀性
- 降低 merge 混亂風險
- 避免分支基底不一致導致衝突放大

---

### 5️⃣ 合併原則（補充建議）

功能完成後：

1. 先將 main merge 進自己的分支
2. 解決衝突並測試
3. 再由該分支 merge 回 main

```
main → 功能分支（同步）
功能分支 → main（合併）
```
